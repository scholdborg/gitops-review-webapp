GitOps Review Webapp

A tiny static site that ships the GitOps way: every change is automatically reviewed and built, and only goes live if every check passes.

Deployed from Git after checks pass

Flödet

En kodändring granskas automatiskt, byggs sedan, och driftsätts till GitHub Pages endast om alla kontroller godkänns. Git är den enda källan till sanning, och GitHub Actions är grinden som avgör om ändringen når den publicerade webbplatsen.

Pipeline

Commit → Review → Build → Approve → Deploy

Each arrow is a gate. If a stage fails, the pipeline stops there and nothing new is deployed — the live site keeps its last good version.

How the workflow works, step by step

1 · On your machine (Claude Code hooks)

  1. You ask Claude to change a file. Before any shell command runs, the guard-dog hook inspects it and blocks dangerous ones (like a forced push or a recursive delete).
  2. Claude writes the change. Right after the edit, the local-review hook re-runs the automated review, and flow-language-guard checks that the Swedish text box is still Swedish. You get feedback in seconds, before anything is committed.
  3. Claude finishes its turn. The deploy-status hook prints a short summary: your branch, whether the working tree is clean, and whether review and build pass.

2 · In the cloud (GitHub Actions)

  1. You commit and push to main. Git is the single source of truth for what should be live.
  2. GitHub Actions runs the gate. It checks out the code, installs dependencies, runs npm run review, then npm run build to assemble the site into dist/.
  3. Deploy only if the gate is green. The deploy job depends on the review-and-build job. If review or build fails, deploy is skipped automatically.
  4. A human approves. The run then pauses on a protected production environment. GitHub emails the required reviewer an Approve/Reject request, and nothing deploys until someone clicks Approve.
  5. GitHub Pages publishes dist/. After approval, the new version becomes the live site at this URL.

What it looks like: passing vs. failing

The review is just a script that prints PASS or FAIL lines and exits with a success or error code. Hooks and GitHub Actions both read that code to decide what happens next.

Successful run

$ npm run review

=== Simple Code Review ===
  PASS  required file present: public/index.html
  PASS  index.html contains a non-empty <title>
  PASS  no obvious secrets found
  PASS  no console.log in src/main.js
  PASS  flow text is Swedish
--------------------------
Result: PASS (7 checks ok)

$ npm run build        # dist/ created
Deploy: published to GitHub Pages ✓

Exit code 0 → the gate is green → GitHub Pages serves the new version.

Failing run

$ npm run review

=== Simple Code Review ===
  PASS  required file present: public/index.html
  FAIL  console.log found in src/main.js
  FAIL  flow text must be written in Swedish
--------------------------
Result: FAIL (2 problems)

# a risky command is also stopped locally:
[guard-dog] BLOCKED: git push --force
Deploy: skipped — checks did not pass

Exit code 1 → the gate is red → deploy is skipped and the live site is unchanged.

A failure is never silent: the failing check is named, the deploy is skipped, and you fix the one thing it points at and push again.

The hooks (your quality gates)

Hooks are small scripts Claude Code runs automatically at points in its lifecycle. They keep edits safe and on-policy before anything is ever committed or deployed.

guard-dog before a shell command
Blocks risky commands before they run, such as rm -rf, git push --force, and npm publish.
local-review after each edit
The smart one. After every edit it (1) re-runs the whole-project review, (2) runs real ESLint on the JavaScript file you just changed (plus text checks like merge markers, long lines, and missing image alt text), and (3) hands the combined summary back to Claude so it can fix issues on the spot. It blocks the turn on serious findings — an ESLint error or a merge marker stops work until it is fixed.
flow-language-guard after each edit
Requires the Flödet text box above to be written in Swedish, and blocks the change if it is not. (That is why this page is partly in Swedish!)
deploy-status when the assistant stops
Prints a GitOps-style summary — branch, working-tree state, and whether review and build pass — at the end of a turn.

Note: the Approve step in the pipeline is not one of these hooks. Hooks run locally in the coding session for fast feedback. Approval is a GitHub Actions environment rule that runs in the cloud after you push: the deploy pauses, the required reviewer gets an email, and nothing goes live until they click Approve. Two complementary gates — a local one while you write, and a remote one that authorizes the real deployment.

The approval gate (human in the loop)

Even after every automated check passes, the deployment waits for a person. The workflow pauses on a protected production environment that has a required reviewer.

  1. Review, lint and build all pass in GitHub Actions.
  2. The run pauses and the deployment shows as Waiting.
  3. GitHub emails the reviewer an approve/reject request (you can also approve from the Actions run page).
  4. On Approve, the deploy job runs and GitHub Pages publishes. On Reject, nothing changes and the live site keeps its last version.

This works on the free plan because the repository is public; for a private repo, environment protection rules need a paid GitHub plan.
